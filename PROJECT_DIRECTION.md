# Напрям розвитку Офіс ПЛЮС

Статус: активне продуктове й технічне джерело правди.

Оновлено: 2026-06-06.

## 1. Короткий контекст для агента

Офіс ПЛЮС — окремий статичний вебзастосунок на GitHub Pages із шістьма локальними редакторами:

- `text/` — текстові документи;
- `tables/` — електронні таблиці;
- `slides/` — презентації;
- `paint/` — растрові малюнки;
- `vector/` — векторна графіка;
- `flowcharts/` — блок-схеми.

Продукт має покривати 80–100% типових редакторських умінь зі шкільної інформатики. Це не система вправ, не LMS і не спроба відтворити весь Microsoft Office.

Обов'язкові обмеження продукту:

- без акаунтів, хмарного бекенду та спільного редагування;
- локальні файли користувача є основним способом тривалого збереження;
- автозбережена браузерна чернетка — лише страховка, а не гарантоване сховище;
- офлайн-робота після першого завантаження;
- простий і зрозумілий інтерфейс для учнів;
- власний спільний shell і дизайн залишаються основою продукту;
- сторонні рушії додаються лише для складних ядер, які невигідно підтримувати самостійно.

Головне архітектурне рішення: **не переписувати весь пакет на сторонній офісний фреймворк**. Розвивати наявні редактори сценаріями, а сторонні бібліотеки підключати точково після окремого пілота.

## 2. Що означає «80–100%»

Ціль стосується редакторських компетентностей, а не всієї шкільної програми з інформатики.

Пакет повинен надійно покривати:

- створення, відкриття, редагування, друк та експорт документів;
- форматування тексту, таблиць, слайдів і графічних об'єктів;
- формули, сортування та базові діаграми;
- створення растрових і векторних зображень;
- створення алгоритмічних блок-схем;
- роботу з локальними файлами без реєстрації.

Пакет не повинен покривати програмування, бази даних, веброзробку, кібербезпеку, тести чи навчальні вправи. Для них потрібні окремі сервіси.

Практичний критерій готовності: учень може виконати типову шкільну редакторську роботу без зовнішнього сервісу, а результат можна відкрити, надрукувати або передати в поширеному форматі.

## 3. Незмінні технічні рішення

### Зберегти

- окремий репозиторій `schooloffice.github.io`;
- статичний клієнтський застосунок і GitHub Pages;
- `office-shell.js`, `office-ui.js`, `UI_TOKENS.css` та спільні UI-контракти;
- локальні `vendor/`-залежності для офлайн-роботи;
- Service Worker;
- стандартні команди `new/open/save/undo/redo`;
- окремі редактори для raster, vector і flowcharts: це різні навчальні моделі;
- локальні формати проєктів для повного відновлення редагованого стану.

### Не робити без нового продуктового рішення

- не додавати акаунти, серверне збереження або collaboration;
- не переносити весь UI на React/Vue;
- не замінювати всі редактори одним великим SDK;
- не додавати ribbon та професійні функції лише заради схожості з Microsoft Office;
- не робити автозбережену браузерну чернетку єдиним способом збереження;
- не вводити build-крок лише тому, що бібліотека популярна.

### Build-крок

Поточний no-build підхід доречний для невеликого офлайн-пакета. Нативні ES modules уже використовуються в `slides/`, тому no-build не означає відмову від модулів.

Build-крок дозволено вводити лише після пілота, який доведе суттєву користь. Він має:

- збирати статичні артефакти для GitHub Pages;
- не ламати офлайн-режим;
- не вимагати сервера під час роботи;
- мати lock-файл, автоматичний dependency audit і відтворюваний CI build;
- не змушувати переписувати спільний shell.

## 5. Оцінка сторонніх кандидатів

### Tiptap / ProseMirror

Вердикт: **не мігрувати Text зараз**.

Переваги:

- зріле транзакційне ядро редагування;
- надійні selection/history/schema;
- Tiptap OSS Editor має MIT-ліцензію і дозволяє зберегти власний UI.

Недоліки для цього продукту:

- міграція змінить внутрішню модель документа й вимагатиме адаптерів;
- A4-пагінація лишається складною окремою задачею;
- Tiptap Pages має обмеження, зокрема щодо таблиць і browser print;
- DOCX/ODT conversion та частина Pages-функцій належать до платної платформи;
- інтеграція практично означає npm/build pipeline.

Тригер для пілота: власний Text стабільно не може забезпечити selection/history/paste після цілеспрямованого циклу виправлень. Пілот має бути окремим прототипом, а не поступовим переписуванням production-коду.

### Univer

Вердикт: **не використовувати як основу всього пакета; розглядати лише пілот Univer Sheets**.

Переваги:

- сильний і активно підтримуваний spreadsheet framework;
- open-source ядро під Apache-2.0;
- значно ширші можливості таблиць, ніж доцільно писати самостійно.

Недоліки для цього продукту:

- великий TypeScript/npm/build стек;
- імпорт/експорт, друк та collaboration віднесені до Univer Pro;
- Pro-функції передбачають комерційну ліцензію та серверну частину;
- Docs і Slides не дають достатньої вигоди для заміни вже наявних редакторів;
- інтеграція може поглинути власний простий UI замість точкової заміни ядра.

Тригер для пілота: потреба в складних формулах, кількох аркушах, великих таблицях і стабільних spreadsheet interactions, які власний grid уже не може підтримувати.

### Luckysheet / FortuneSheet

Вердикт: **не обирати як нову основу**.

Luckysheet архівований на користь Univer. FortuneSheet може бути корисним для експериментів, але нове стратегічне ядро не слід будувати на менш очевидному довгостроковому напрямі, якщо є активніший наступник.

### HyperFormula

Вердикт: **не інтегрувати без свідомого ліцензійного рішення**.

HyperFormula технічно підходить як формульний рушій, але open-source варіант використовує GPLv3; альтернатива — комерційна ліцензія. GPL може вимагати поширення сумісного похідного продукту на умовах GPL. Перед пілотом потрібна юридична оцінка та рішення щодо ліцензії самого Офіс ПЛЮС.

### PptxGenJS

Вердикт: **найкращий точковий кандидат для Slides**.

PptxGenJS має MIT-ліцензію, працює у браузері та генерує `.pptx`. Його варто використовувати як export adapter поверх власної моделі Slides, а не як редактор.

Пілот повинен перевірити текст, зображення, базові фігури, таблиці, діаграми, українські шрифти та відкриття результату в PowerPoint/LibreOffice/Google Slides.

### Mermaid

Вердикт: **додаткова сумісність для Flowcharts, не заміна редактора**.

Mermaid має MIT-ліцензію. Корисні сценарії: експорт простої блок-схеми в Mermaid syntax та імпорт обмеженої підмножини. Візуальний drag editor лишається основним.

## 6. Оптимальний порядок розвитку

### Етап 0. Негайна гігієна

1. Визначити ліцензію власного репозиторію або явно позначити його як proprietary/all-rights-reserved.
2. Додати `THIRD_PARTY_NOTICES.md` і локальні тексти ліцензій для всього `vendor/`.
3. Зафіксувати назви, версії, джерела та SHA-256 сторонніх bundle-файлів.
4. Виправити пошкоджене кодування активної документації.
5. Прибрати застарілі посилання на старий шлях `/office`.
6. Додати dependency-update та security-review процес.

### Етап 1. Надійність базових сценаріїв

Спочатку не додавати великі функції. Для кожного редактора автоматизувати:

- boot;
- new/open/save;
- undo/redo;
- імпорт валідного й невалідного файла;
- експорт і повторне відкриття;
- друк або print-preview;
- autosave/restore та quota failure;
- keyboard-only базовий сценарій.

Найвищий пріоритет безпеки: Vector, Slides і Paint project import; найвищий пріоритет надійності: Text selection/pagination і Tables workbook/formulas.

### Етап 2. Локальне сховище

Зберігати `localStorage` лише для дрібних налаштувань і теми. Для великих чернеток із зображеннями перейти на IndexedDB, яке підтримує structured data та Blob.

Правила:

- чернетка завжди позначається як негарантована;
- користувачеві регулярно пропонується завантажити файл;
- помилки quota/eviction не повинні ламати редактор;
- усі редактори мають окремі versioned storage keys;
- має бути команда очищення локальних чернеток.

### Етап 3. Функціональне покриття школи

Пріоритет за навчальною цінністю:

1. Text: стабільні сторінки, списки, таблиці, зображення, headers/footers за потреби, друк.
2. Tables: базові формули, абсолютні/відносні посилання, кілька аркушів за підтвердженої потреби, діаграми.
3. Slides: теми, макети, вирівнювання, таблиці/діаграми, PPTX export.
4. Paint: crop/resize, шари або спрощені об'єкти, текст, прозорість.
5. Vector: групування, transform, криві/paths, SVG import/export.
6. Flowcharts: надійні зв'язки, вирівнювання, auto-layout, Mermaid compatibility.

### Етап 4. Пілоти сторонніх ядер

Пілоти проводяться окремо й мають вимірювані критерії. Не починати два великі пілоти одночасно.

Рекомендований порядок:

1. PptxGenJS export adapter.
2. Mermaid import/export subset.
3. Univer Sheets proof of concept лише після підтвердженої межі власних Tables.
4. Tiptap proof of concept лише після підтвердженої межі власного Text.

## 7. Правові ризики

Цей розділ не є юридичною консультацією.

### Поточні проблеми

- У репозиторії немає кореневого `LICENSE`, тому права користувачів і контриб'юторів не визначені явно.
- У `vendor/` є сторонні bundle-файли без окремого реєстру версій і notices.
- MIT, BSD, Apache, MPL, SIL OFL та CC BY мають різні вимоги щодо збереження copyright/license notices.
- Font Awesome Free використовує різні ліцензії для коду, шрифтів та іконок.
- Неможливо безпечно оновлювати або перевіряти залежності, якщо невідомі точні версії bundle-файлів.

### Мінімальний правовий baseline

- обрати ліцензію власного коду;
- додати `THIRD_PARTY_NOTICES.md`;
- зберігати тексти ліцензій разом із vendored залежностями;
- не видаляти copyright headers;
- документувати джерело й версію кожного bundle;
- перед GPL/AGPL/SSPL або commercial/pro залежністю робити окреме рішення;
- не використовувати торгові марки так, щоб створювати враження офіційного зв'язку з Microsoft, Google чи іншими продуктами.

Поточні залежності, які треба інвентаризувати першими:

- Chart.js — MIT;
- docx — MIT;
- DOMPurify — Apache-2.0 або MPL-2.0;
- Font Awesome Free — MIT для коду, SIL OFL для шрифтів, CC BY для SVG/JS icons;
- html2canvas — MIT;
- html2pdf.js та його bundled залежності — MIT, але треба врахувати notices вкладених залежностей;
- Mammoth — BSD-2-Clause.

## 8. Безпекові ризики

Хмарних акаунтів немає, тому головна поверхня атаки — **відкриття недовірених локальних файлів і рендеринг їхнього вмісту**.

### P0

- Додати schema validation, type allowlists, межі кількості об'єктів, довжин рядків і числових координат для Vector, Slides, Paint та Tables workbook.
- Не вставляти дані з project JSON у SVG/HTML через шаблонні рядки без escaping або DOM API.
- Залишити DOMPurify обов'язковим для DOCX/RTF/HTML-вмісту; не покладатися на fallback як рівноцінний захист.
- Обмежити розмір імпортованих файлів, data URLs, зображень, кількість слайдів/об'єктів і глибину вкладеності.
- Заборонити або явно підтверджувати зовнішні image URLs у Slides: вони розкривають IP, залежать від CORS і ламають офлайн.

### P1

- Додати Content Security Policy для GitHub Pages-сумісного статичного застосунку.
- Для всіх `target="_blank"` використовувати `rel="noopener noreferrer"`.
- Перевіряти MIME, extension і фактичну структуру імпортованих файлів.
- Додати негативні browser tests для XSS-подібних payload, величезних JSON і пошкоджених файлів.
- Не логувати вміст учнівських документів.
- Переглядати Service Worker cache/version під час кожного релізу; старий кеш не повинен утримувати вразливі bundle-файли.

### Наявні позитивні рішення

- Text очищує імпортований DOCX HTML через DOMPurify.
- Tables використовує власний parser формул замість `eval`.
- Flowcharts уже має обмеження кількості shapes/connections і нормалізацію проєкту.
- Залежності локалізовані, тому робота не залежить від стороннього CDN.

## 9. Тестова стратегія

Поточні static audit і browser smoke треба зберегти, але вони перевіряють переважно shell та частину поведінки.

Для кожної нової функції потрібен тест на рівні її ризику:

- pure/domain test для parser, normalization, formula або serializer;
- browser behavior test для selection, pointer, file import та export;
- round-trip fixture: save -> open -> порівняння суттєвого стану;
- malformed/untrusted fixture для кожного формату імпорту;
- ручна перевірка друку та файлів у зовнішніх програмах перед релізом.

Обов'язковий CI baseline:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File tests\run-tests.ps1
powershell -NoProfile -ExecutionPolicy Bypass -File tests\run-browser-smoke.ps1
git diff --check
```

## 10. Правила прийняття нової залежності

Перед додаванням бібліотеки відповісти письмово:

1. Яку конкретну складну проблему вона прибирає?
2. Чому дешевше інтегрувати й підтримувати її, ніж допрацювати наявний код?
3. Чи працює вона повністю локально й офлайн?
4. Чи потрібні сервер, платний Pro-модуль або build-крок?
5. Яка ліцензія бібліотеки та її транзитивних залежностей?
6. Який розмір bundle і вплив на слабкі шкільні пристрої?
7. Як вона інтегрується з власним shell, файлами й тестами?
8. Який план виходу, якщо проєкт покинуть або змінять ліцензію?

Якщо немає чітких відповідей, залежність не додається.

## 11. Definition of Done для редактора

Редактор вважається достатнім для базового шкільного використання, коли:

- основні сценарії можна виконати клавіатурою і мишею;
- відкриття пошкодженого або недовіреного файла не ламає застосунок;
- save/open round-trip не втрачає суттєвий стан;
- експортований поширений формат відкривається у принаймні двох зовнішніх програмах;
- друк має передбачуваний результат;
- редактор працює офлайн після першого завантаження;
- помилка локального сховища не знищує поточну роботу;
- static audit і browser smoke проходять;
- сторонні залежності мають задокументовані ліцензії та версії.
